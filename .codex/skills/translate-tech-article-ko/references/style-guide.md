# Translation Style Guide

기준 샘플:

- `src/translations/two-pizza-team.md`
- `src/translations/effective-context-engineering-for-ai-agents.md`

이 skill은 위 두 번역본의 공통된 문체와 편집 습관을 재현해야 한다.

## Core Voice

- 차분하고 설명적인 기술 블로그 문체를 유지한다.
- 직역보다 자연스러운 한국어 독해감을 우선하되, 의미 손실이 생기면 안 된다.
- 과장, 마케팅 문구, 감탄형 표현, 불필요한 구어체를 피한다.
- 문장 끝은 주로 `-합니다`, `-합니다.`, `-뜻합니다`, `-가리킵니다`처럼 단정적이고 안정적인 서술형으로 맞춘다.

## Structure

- frontmatter의 `title`은 기본적으로 원문의 영문 제목을 유지한다.
- 문서 첫머리에는 원문 링크 안내 bullet을 반드시 둔다.
- 이 bullet은 frontmatter 다음에 바로 와야 하며, 본문 첫 문단보다 앞서야 한다.
- 가능하면 `- 원문은 [Source Name의 원 글](URL)을 확인해주세요.` 형식을 유지한다.
- 원문이 서론 문단으로 시작하면 한국어 번역도 짧은 도입 문단을 유지한다.
- 헤딩 언어는 기사 성격에 맞춰 정한다.
  - 개념 설명형 짧은 글은 한국어 소제목이 잘 맞는다.
  - 원문 헤딩이 기술 용어 자체이거나 추적 가능성이 중요하면 영문 헤딩을 유지한다.
- 한 문단이 과하게 길어지면 한국어 가독성에 맞춰 재단락한다.

## Terminology

- 핵심 기술 용어는 한국어 번역만으로 의미가 흐려질 때 첫 등장에 영어를 병기한다.
  - 예: `Context`, `AI agent`, `cycle time`, `capability`
- 이미 한국어 업계 용례가 자연스러운 용어는 한국어를 우선하고 필요 시 영어를 한 번만 덧붙인다.
- 모든 명사를 기계적으로 병기하지 않는다.
- 제품명, 프레임워크명, 논문명, 원칙 이름은 원문 표기를 존중한다.

## Rendering Rules

- 링크는 유지한다.
- 의미 있는 이미지와 도식은 유지한다. 원문에 없는 alt 텍스트나 캡션은 추가하지 않는다.
- 목록, 표, 인용, 코드 블록 구조는 유지한다.
- 코드와 식별자는 번역하지 않는다.
- 필요할 때만 짧은 괄호 설명을 넣고, 원문에 없는 장황한 해설은 추가하지 않는다.

## Rewrite Rules

- 영어의 압축된 문장을 그대로 옮기지 말고 한국어 문장 순서로 다시 풀어 쓴다.
- 원문의 논리 관계는 유지하되 접속사를 남발하지 않는다.
- 지나치게 축약하지 않는다. 번역본은 요약문이 아니라 게시 가능한 전체 번역이어야 한다.
- 원문에 없는 주장, 판단, 예시는 넣지 않는다.

## Calibration

- 먼저 이 문서를 기준으로 작업한다.
- 번역 대상이 아래 중 하나에 해당하면 샘플 원문을 직접 다시 연다.
  - 개념 사전형 짧은 글
  - 긴 엔지니어링 에세이
  - 이미지/도식이 많은 글
  - 용어 선택이 애매한 글
